Тестовое задание на вакансию junior-python разработчик. Приложение реализует систему аутентификации/авторизации пользователей с использованием прав доступа

Технологии - Python 3.12, FastAPI, SQLAlchemy, PostgreSQL, Alembic, JWT, Docker
-----------------------------------------------------------------------------------
Способ аутентификации(реализована через JWT токены):
/login 
пользователь отправляет логин(email)/пароль на эндпоинт POST /auth/login 
-берем данные из формы, проверяем зарегистрирован ли пользователь, пароль
-генерируем jwt-токен, в payload добавляем его id, подписываем симметричным ключом
-swagger поддерживает возможность аутентификации через кнопку authorize, и дает возможность тестирования в автоматической документации автоматически подставляя токен в заголовке Authorization

При обращении к защищенным эндпоинтам
-из заголовка извлекается токен и декодируется
-по user_id из payload загружаем пользователя из базы, проверяем активность (is_active=Rtue)
Ошибки:
401 - Not authenticate выбрасывает FA, т.к в заголовке не передан токен
401 - Ошибка авторизации, выбрасываем в случае если пользователя нет в БД или введенный пароль не совпадает
403 - Ошибка, в случае если активность(is_active=False)

/logout , POST auth/logout
является условным эндпоинтом, поскольку JWT хранится на стороне клиента

Удаление реализовано в мягком варианте, при удалении DELETE /users/me
устанавливается статус is_active=False после чего нельзя пройти повторную аутентификацию

при обновлении есть возможность обновить поля - имя/фамилия/пароль 
PATCH /users/me 
после декодирования токена, загрузки пользователя и проверки статуса выполняется обновление данных 
----------------------------------------------------------------------
Разграничение прав доступа

таблица roles - пользовательские роли
id | name 

содержит следующие роли - admin, user, manager
таблица business_elements - описывает бизнес-сущности приложения, к которым можно получать доступ
id | name
содержит сущности - products, category, access_rules

таблица access_role_rules - правила доступа ролей пользователей к бизнес-сущностям
Содержит поля - role_id,element_id,read_permission,read_all_permission,create_permission,update_permission,update_all_permission,delete_permission,delete_all_permission

Правила определяют что пользователь может делать с объектами
При обращении к защищенному эндпоинту
-извлекается токен из заголовка, достаем user_id
-делаем запрос в базу, используем технологии подгрузки отношений selectinload/joinedload (загружается роль пользователя и его права доступа)
-проверяем разрешено ли пользователю действие
Ошибки
-если пользователь не залогинен - 401 ошибка,
-если пользователь определен, но ресурс не доступен - 403 ошибка

Для демонстрации работы системы реализованы mock endpoints (сущности product/category), которые имитируют бизнес-ресурсы.
Они возвращают тестовые данные и проверяют права доступа пользователя

Для запуска проекта необходимо:
-используя команду git clone <repo_url> . скопироват все файлы из удаленного репозитория
-создать и активировать виртуальное окружение 
-внутрь виртуального окружения установить систему контроля зависимостей poetry "pip install poetry"
-выполнить установку всех необходимых зависимостей для корректной работы приложения "poetry install"
-запустить докер-контейнер с базой "docker compose up -d"
-выполнить миграции "alembic upgrade head"
-выполнить скрипт для заполнения базы тестовыми данными "python -m app.database.seed_data"
-запустить локальный вб-сервер uvicorn для тестирования
